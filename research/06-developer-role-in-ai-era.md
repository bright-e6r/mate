# AI가 코드를 쓰는 시대, 개발자의 진짜 역량이 드러난다

> https://evan-moon.github.io/2026/02/10/developer-in-ai-era/
> 작성자: Evan Moon

## ChatGPT 이후 3년

- 처음 ChatGPT가 등장한 2022년 말, 할루시네이션 이슈도 많고 출력 퀄리티도 좋지 않았음
- 겨우 3년 만에 개발자의 하루가 상당히 많은 부분이 변화함
- 과거: 아침에 출근하면 에디터를 열고 빈 파일에 커서를 놓는 것으로 하루를 시작
- 현재: AI에게 맥락을 전달하고, 생성된 코드를 읽고, 고치고, 다시 요청하는 것이 하루의 많은 부분
- AI 코딩 에이전트의 성능은 단순 보조 수준을 넘어섬 — 프롬프트만 잘 제공하면 사람이 작성한 것과 구별 어려운 코드를 내놓음
- 앞으로 변화 속도는 더 빨라질 가능성이 높으며, 개발자의 역할도 팀과 조직의 맥락에 따라 다르게 재편될 것

## 작성자에서 의사결정권자로

- 코드를 통해 제품을 생산하는 행위는 직접적인 코드 작성에서 점점 **"코드에 대한 판단"**으로 이동하고 있음
- 판단이란 단순히 AI 출력이 의도대로 됐는지 확인하는 것이 아니라 **비즈니스 의도가 기술적 구현으로 올바르게 번역됐는지 검증하는 과정**

### 코드에 대한 책임을 질 사람이 필요하다

- AI는 법적으로 인격체가 아니기 때문에 책임을 질 수 없음
- 아무리 AI가 작성한 코드라도 결국 누군가는 "책임"을 져야 하며, 그것은 코드를 리뷰하고 승인한 개발자와 그 조직
- EU AI Act (2024): 고위험 영역에서 AI 시스템에 인간 감독을 의무화
- EU 제품책임 체계를 디지털 제품/소프트웨어까지 포괄하려는 논의 진행 중
- 자율주행차, 의료 AI 진단, 알고리즘 트레이딩 등 실제 사례에서도 **자동화가 고도화될수록 책임 구조는 인간 쪽으로 귀속**되는 경향
- 개발자의 역할이 사라지기보다는 **코드를 작성하는 사람에서 "코드를 리뷰하고 승인 도장을 찍는 사람"으로 변화**

## AI 시대에 개발자에게 요구되는 역량

- 중요한 것은 프롬프트를 잘 작성하는 것이 아니라, 퀄리티 기준을 충족하지 않는 코드가 출력됐을 때 **문제를 감지할 수 있는 능력**
- AI 시대에 요구되는 역량은 지금 좋은 개발자에게 요구되는 역량과 크게 다르지 않음
- 코드에 최종 승인 도장을 찍는 사람이 존재한다는 것은 여전히 **"인간이 이해하기 쉬운 코드"를 작성해야 한다는 것**을 의미

### 장기 변경 비용을 예측하는 역량

- AI는 "작동하는 코드"를 만드는 데 최적화되어 있음 — 학습 데이터에서 가장 빈번하게 등장하는 패턴을 재현
- **지금 당장 작동하는 코드와 6개월 뒤에도 유지보수가 쉬운 코드는 전혀 다른 기준**
- 소프트웨어의 품질은 즉시 드러나지 않음 — 나쁜 설계의 비용은 코드를 작성한 시점이 아니라 **그 코드를 변경해야 하는 시점**에 발생

### 코드를 다각도로 평가하는 역량

- 기능적 정확성은 테스트로 어느 정도 검증 가능
- 더 까다로운 건 **구조적 품질**: 모듈의 책임 범위, 의존성 방향, 인터페이스의 변경 유연성 — 자동화된 테스트로는 잡기 어려움
- 그 너머에 성능적 함의(데이터가 10배 늘었을 때), 보안적 측면(입력 검증, 권한 확인)도 존재
- AI가 수 초 만에 수백 줄의 코드를 생성하는 환경에서 **코드 생산 속도와 리뷰 역량 사이의 균형이 쉽게 깨짐**
- 코드 생산은 빨라졌지만 AI가 뱉은 코드를 리뷰하는 과정이 **제일 병목**

### 추상화 역량

- 추상화란 복잡한 시스템에서 **무엇을 감추고 무엇을 드러낼지, 어디에 경계를 그을지 결정하는 능력**
- AI도 인터페이스를 정의하고 클래스를 나누고 모듈을 분리할 수 있음 — 형식적으로는 인간보다 나을 수도 있음
- 하지만 AI의 추상화는 **학습 데이터의 통계적 평균**에 기반
  - 실제 소프트웨어 설계는 정답을 맞히는 게임이 아니라 **한정된 자원과 불확실한 미래 속에서 무엇을 포기할지 결정하는 트레이드오프의 영역**
  - AI는 코드 내부의 정합성에 능숙하지만 **코드 외부의 맥락까지 고려하여 "어디에 경계를 그을지" 결정하는 것은 어려워함**
- AI가 만든 코드의 위험: 겉보기에 좋아 보임 — 파일이 적절히 나뉘어 있고, 네이밍도 관례를 따르고, 패턴도 익숙함
  - 문제는 **실제로 변경이 필요한 시점**에 발견됨
  - 결제 수단을 하나 추가하려는데 "깔끔하게 나뉘어 있던" 구조의 여기저기를 동시에 수정해야 한다는 걸 깨닫게 됨
- 추상화 부족(거대한 컴포넌트)과 추상화 과잉(간단한 차트에 커스텀 훅 세 개와 컨텍스트 프로바이더) 모두 **현재 프로젝트의 복잡도 수준에 맞지 않는다는 점에서 같은 문제**
  - **적절한 추상화란 현재 상황에서 딱 필요한 만큼의 구조를 만드는 것**, 이 "만큼"을 판단하는 건 맥락을 아는 사람만 할 수 있음

### 암묵지를 명시화할 수 있는 역량

- 추상화 역량이 "어디에 경계를 그을지 아는 것"이라면, 암묵지를 명시화하는 것은 **"왜 거기에 경계를 그어야 하는지 설명할 수 있는 것"**
- 좋은 코드와 나쁜 코드를 구분하는 감각은 코드를 많이 읽고 쓰면서 자연스럽게 체득 — 보통 "뭔가 이상한데"라는 직감으로 시작
- 이 암묵지를 언어로 만드는 능력이 AI 시대에 특히 중요:
  - "뭔가 이상한데" → "이 함수는 두 가지 책임을 가지고 있다", "이 인터페이스가 변경에 취약하다"
  - 이렇게 구체적인 언어로 바꿀 수 있어야 AI에게 올바른 명령을 내릴 수 있음
- 중요한 건 few-shot이나 chain-of-thought 같은 형식적 프롬프트 기법이 아님
  - 그 아래에 있는 능력: **무엇을 만들어야 하는지 명확하게 정의하고, 어떤 맥락이 중요한지 판단해서 전달하는 것**
- 이 역량들의 유효기간은 도구 숙련도보다 훨씬 긺
  - 도구 숙련도의 수명은 도구의 교체 주기에 묶여 있음 (jQuery→React, Webpack→Vite)
  - 설계 판단과 암묵지 언어화 역량의 수명은 **소프트웨어의 본질적 복잡성이 존재하는 한 유효**
  - 결제 시스템이 복잡한 이유는 프레임워크 때문이 아니라 결제 도메인 자체가 복잡하기 때문

## 의도적 수련 설계의 필요성

- 코드를 직접 작성하는 시간이 줄어드는데, 추상화나 설계 감각은 어떻게 기르는가?
- "도구가 할 수 없는 것에 집중하라"고 말하면서 정작 그것을 기를 기회가 줄어드는 모순적 상황
- 해법은 두 층: 업무 흐름에서 AI에게 넘기지 말아야 할 지점을 지키는 것 + 설계 감각을 업무 밖에서 의도적으로 유지하는 것

### AI에게 넘기지 말아야 할 두 지점: 설계와 리뷰

- **설계:** 프롬프트를 치기 전에 "이 모듈이 외부에 노출할 인터페이스는 무엇인가", "책임의 경계는 어디에 그을 것인가"를 먼저 정의
  - 그 후 구현을 AI에게 위임하면, AI가 만들어낸 결과물을 **자신의 설계 결정과 비교**할 수 있게 됨
  - 이 과정이 부재하면 단순히 AI가 출력한 코드를 소비하게 됨 — 소비하는 것과 자신의 판단을 기준으로 평가하는 것은 완전히 다른 경험
- **리뷰:** 업무 흐름 안에서 자동으로 주어지는 역량 성장 기회
  - AI에게 PR 리뷰를 맡기고 승인해버리는 습관은 **체육 시간에 운동장에 나가서 벤치에만 앉아 있는 것**과 같음
  - 코드 리뷰는 요구사항, 설계 의도, 비즈니스 맥락을 동시에 고려하면서 코드를 읽는 과정 — 이 과정 자체가 설계 감각을 유지하는 훈련

### 의도적으로 직접 짜보는 시간

- 설계 감각 자체가 **구현의 고통을 알아야 생김** — 어떤 구조가 변경에 취약한지를 직접 손으로 겪어봐야 "이 추상화가 잘못됐다"는 직감이 생김
- 주니어의 경우 특히 심각: 경험이 적은 상태에서 AI가 생성한 코드를 리뷰하는 건 **운전을 배우는 중인 사람에게 자율주행차의 판단을 평가하라고 하는 것**과 비슷
- 미래의 개발자에게 코딩하는 역량은 매일 하는 업무가 아니라 **판단력을 유지하기 위한 훈련**이 될 가능성이 높음
- 사이드 프로젝트나 개인 학습에서는 의도적으로 AI를 내려놓고 처음부터 끝까지 직접 구현해보는 시간을 만드는 게 좋음

### 두 가지를 관통하는 태도: "왜"를 언어화하는 연습

- "뭔가 이상한데"라는 직감을 그냥 넘기지 않고 **왜 그렇게 느꼈는지 명확한 말로 표현**할 수 있을 때까지 다듬는 것
- 직감은 나만 아는 것이고 언어는 쓸 수 있는 것
  - 언어가 되어야 AI에게 정확한 지시를 내릴 수 있고, 팀원에게 설명할 수 있고, 다음에 비슷한 패턴을 만났을 때 알아챌 수 있음
- "왜 이 구조를 선택했을까", "다른 구조였다면 어떤 트레이드오프가 있었을까"를 스스로에게 물어보는 습관
  - 이 습관이 있는 개발자와 없는 개발자 사이의 격차는 시간이 갈수록 벌어질 것

## 결국 본질은 변하지 않았다

- 생산과 검수를 모두 인간이 하던 시대에서 **생산은 AI가 하고 검수는 인간이 담당**하는 시대로 변화
- 이 변화가 기술적 한계가 아닌 **책임 소재라는 사회적 문제**에서 비롯됨
- 리뷰 승인을 잘 찍으려면:
  - 작동하는 코드와 오래 살아남는 코드를 구분하는 눈
  - 맥락에 맞는 추상화를 판단하는 설계 감각
  - 그 감각을 언어로 만들어 정확한 방향을 잡는 역량
- Fred Brooks (1986): 소프트웨어의 복잡성을 두 가지로 나눔
  - **우발적 복잡성**: 도구의 한계에서 오는 것 — AI가 해결하는 부분 (보일러플레이트, 반복 패턴, 문법 오류)
  - **본질적 복잡성**: 문제 자체에 내재된 것 — 비즈니스 요구사항의 모호함, 상충하는 설계 목표, 미래 변경 방향의 불확실성 — **AI가 아무리 발전해도 사라지지 않음**
- AI의 성격이 이전 도구들과 달라도 인간이 판단과 책임의 주체로 남아 있는 한 그 판단에 필요한 역량의 본질은 바뀌지 않음
- 오히려 코드 생산이 자동화될수록 **그 생산물을 검수하는 판단력의 비중이 부각**될 것

> "물이 빠지면 비로소 누가 발가벗고 수영을 하고 있었는지 알 수 있을 것"
